# 매매 로직 제안 → 코드 반영 에이전트 설계서 및 고려사항

> 목적: Gemini·ChatGPT와 구현 구체화를 위한 논의용 문서.  
> 현재 프로젝트: 업비트 스캘핑 대시보드 (Node.js, Discord 봇, PM2 배포)

---

## 1. 개요

### 1.1 목표

- **입력**: Discord 버튼 「💡 매매 로직 수정안 제안」에서 나온 **자연어 제안 텍스트** (예: "entry_score_min을 55로 낮추세요", "max_spread_pct 상향 검토").
- **출력**: 해당 제안을 **실제 코드/설정에 반영**하고, 필요 시 **서버 재시작**까지 수행하는 자동화(에이전트).
- **제약**: 매매 로직·자금에 직접 영향이 있으므로 **안전 장치**(확인 단계, 롤백, 감사 로그)가 필요함.

### 1.2 현재 구현 상태

| 구분 | 내용 |
|------|------|
| 제안 생성 | `lib/diagnoseNoTradeAnalyzer.js`의 `buildSuggestSummary(diagnosticsStore)` — 로컬 규칙 기반, Gemini 미사용 |
| 제안 노출 | Discord Embed으로 5줄 이내 텍스트 출력 |
| 데이터 소스 | 최근 12시간 거래/거절 로그, 프로필(entry_score_min, rsi_oversold 등), 잔고·엔진 상태 |
| 수정 대상 후보 | `config.default.js`, `lib/StrategyManager.js`, `lib/scalpEngine.js` 등 프로필/설정 관련 코드 |

## 2. 설계 옵션

### 2.1 제안 표현 방식

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 자연어만** | 현재처럼 "entry_score_min을 55로 낮추면…" 형태의 문장만 출력 | 구현 간단, 사용자 읽기 좋음 | 에이전트가 파싱·매핑해야 함, 오해 가능성 |
| **B. 구조화 출력** | 제안을 JSON 등으로 보조 출력 (예: `{"entry_score_min": 55}`) | 에이전트가 적용 로직 작성하기 쉬움 | 제안 문장과 이중 관리, 제안 문구가 제한적 |
| **C. 하이브리드** | 사용자용은 자연어, 에이전트용은 별도 구조화 채널(파일/API)에 저장 | 읽기 좋음 + 자동 적용 정확 | 구조화 스키마·채널 설계 필요 |

**논의 포인트**: 자동 적용 비율을 높이려면 B 또는 C, 사람이 제안만 읽고 Cursor에 복붙하는 수준이면 A만으로도 가능.

---

### 2.2 에이전트 실행 주체

| 옵션 | 설명 | 고려사항 |
|------|------|----------|
| **Cursor Rule + 사용자 프롬프트** | 사용자가 제안 텍스트를 복사해 Cursor에 붙여넣고, 고정된 프롬프트로 "이 제안대로 수정해줘" 요청 | 구현 없음, 제어는 사용자. 재시작은 수동. |
| **Cursor Skill (SKILL.md)** | "매매 로직 제안 적용" 전용 스킬로 단계(파싱 → 수정 대상 결정 → diff 제시 → 적용) 정의 | 재사용·일관성 확보. Skill 작성·유지보수 필요. |
| **봇 + 로컬 스크립트** | Discord에서 "적용해줘" 버튼 클릭 → 대시보드 API가 제안을 받아 스크립트 실행 → Cursor API 또는 CLI로 변경 요청 | 완전 자동에 가깝지만 Cursor API/CLI 의존성·보안 이슈 검토 필요. |
| **봇 + 템플릿 패치** | 제안을 미리 정의된 "프로필 패치" 집합으로 매핑(예: "RSI 완화" → entry_score_min: 55)하고, 설정 파일만 덮어쓰기 | 구현 단순, 표현력은 제한적. |

**논의 포인트**: "서버까지 구현"을 "재시작 명령 안내" 수준으로 할지, "재시작 스크립트 자동 실행"까지 할지에 따라 위 옵션 선택이 갈림.

---

### 2.3 수정 허용 범위 (화이트리스트)

에이전트가 변경해도 된다고 명시할 대상만 두는 것이 안전함.

- **파일**
  - `config.default.js` (DEFAULT_PROFILE, RACE_HORSE_OVERRIDES 등)
  - `lib/StrategyManager.js` (프로필 읽기/쓰기)
  - (선택) `lib/scalpEngine.js` — 상수·게이트 로직은 읽기 전용, 프로필 관련만 간접 변경
- **금지**
  - API 키·시크릿, `.env`, 주문/체결 로직, DB 스키마 등
- **논의 포인트**: "진입 점수 완화"가 `entry_score_min` 한 필드만 바꾸면 되는지, 다른 파일(예: EntryScoreCalculator 가중치)까지 건드려야 하는지 프로젝트 구조에 맞게 정리 필요.

## 3. 안전 장치 및 고려사항

### 3.1 적용 전 확인

- **항상 diff 먼저**: 실제 파일 수정 전에 변경 예정 내용(diff)을 보여주고, 사용자 확인 후 적용.
- **드라이런 모드**: "적용하지 말고, 어떤 파일을 어떻게 바꿀지만 보고해줘" 옵션.
- **권한**: "매매 로직 제안 적용" 실행은 ADMIN만 (이미 Discord 쪽은 관리자 제한됨; Cursor/스크립트 쪽도 동일 정책 적용 여부 논의).

### 3.2 롤백 및 감사

- **변경 전 스냅샷**: 적용 직전에 수정 대상 파일의 백업 또는 git commit 생성.
- **감사 로그**: "언제, 어떤 제안 텍스트로, 어떤 파일을 어떻게 바꿨는지"를 로그/DB에 기록 (선택).
- **롤백 방법**: git 사용 시 `git checkout -- <file>` 또는 백업 파일 복원 절차를 설계서에 명시.

### 3.3 서버 반영

- **재시작 정책**: 코드 변경 후 반영을 위해 `node server.js` 재실행 또는 `npm run pm2:main:restart` 필요.
- **자동 재시작 여부**: 
  - 자동 실행 시: PM2/프로세스 매니저에 의존, 실패 시 알림 필요.
  - 수동 안내만 할 경우: "다음 명령으로 재시작하세요: ..." 문구 출력으로 끝.

## 4. 기술적 고려사항

### 4.1 제안 → 설정 매핑

- 자연어 제안과 실제 설정 키/파일 대응표가 있으면 에이전트 구현이 수월함.
- 예시:
  - "entry_score_min을 낮추세요" → `config.default.js` 또는 StrategyManager 프로필의 `entry_score_min`
  - "max_spread_pct를 올리세요" → `max_spread_pct`
  - "RSI 진입 기준 완화" → `rsi_oversold` 또는 `entry_score_min` (프로젝트 정의에 따름)

### 4.2 프로필 적용 경로

- 현재: StrategyManager가 프로필 보관, scalpEngine은 `getProfile()`로 참조.
- 설정 파일만 수정하면 재시작 전까지 반영 안 됨; 재시작 시 config/StrategyManager를 다시 읽는지 확인 필요.
- **논의 포인트**: 런타임에 API로 프로필만 바꾸는 경로(예: `/api/profile` PATCH)를 두고, 에이전트는 "파일 수정" 대신 "API 호출"만 할지도 선택지.

### 4.3 Cursor 연동

- **Cursor API/CLI**: 공식적으로 "제안 텍스트를 주고 코드 수정 요청"을 자동화하는 API가 있는지, 제한사항은 무엇인지 확인 필요.
- **Rule/Skill만 사용**: API 없이, 사람이 Cursor 채팅에 제안을 붙여넣고 Rule/Skill이 정한 대로 동작하도록 하는 방식이면, "에이전트"는 "사람 + Cursor + 고정 프롬프트" 조합으로 정의할 수 있음.

## 5. 구현 단계 제안 (논의용)

1. **Phase 1**: 제안 출력 구조화 (선택)  
   - `buildSuggestSummary`가 자연어와 함께 구조화 데이터(예: JSON)를 내보내거나, 별도 API/파일로 저장.
2. **Phase 2**: Cursor Rule 또는 Skill 작성  
   - "매매 로직 제안 적용" 시 수정 가능 파일, 적용 전 diff 필수, 롤백 안내 문구 등을 문서화.
3. **Phase 3**: 사용자 플로우 정리  
   - Discord 제안 확인 → (선택) 구조화 데이터 복사 또는 API 조회 → Cursor에 프롬프트 + 제안 붙여넣기 → diff 확인 → 적용 → 재시작 안내/실행.
4. **Phase 4**: (선택) Discord에서 "적용 요청" 버튼 추가  
   - 클릭 시 대시보드가 제안을 저장하고, 로컬 스크립트 또는 Cursor 연동이 그걸 읽어 처리하는 방식. Cursor API/CLI 정책 확인 후 설계.

## 6. Gemini·ChatGPT와 논의할 때 넣으면 좋은 질문

- 우리 제안 문장(한국어, 5줄 이내)을 **구조화 데이터(JSON)** 로 변환하는 규칙 또는 프롬프트를 어떻게 설계하면 좋을지?
- "entry_score_min 55로 조정", "max_spread_pct 소폭 상향" 같은 문구를 **특정 설정 키·파일·값**으로 매핑할 때, 오해를 줄이는 프롬프트나 스키마 예시는?
- Cursor에서 **매매 로직 제안을 받아 코드만 수정**(재시작은 사용자)하는 Rule/Skill 문장 예시를 하나 만들어 줄 수 있는지?
- **안전하게** "제안 적용 → diff 확인 → 적용 → 재시작 안내" 단계를 자동화하려면, 각 단계에서 어떤 입력·출력·검증을 두는 게 좋은지?
