| 항목 | 내용 |
|------|------|
| **services-status reason** | boolean만 반환하던 응답에 `details: { apiServer, marketBot, engine }.reason` 추가. timeout / connection refused / unreachable / stopped / unavailable 구분. |

## 4. 실제 코드 수정안

### 4-1. /api/services-status reason 확장 (api-server)

- **deriveReason**: proxy 실패 시 `data.error` 내용으로 timeout / connection refused / unreachable / unavailable 정규화.
- **응답**: 기존 `apiServer`, `marketBot`, `engineRunning` 유지하고 `details: { apiServer: { reason }, marketBot: { reason }, engine: { reason } }` 추가.
- **engine reason**: engine ok이면 `status !== 'RUNNING'`일 때 `'stopped'`, 실패 시 deriveReason(engineR).

```ts
function deriveReason(r: { ok: boolean; data?: { error?: string }; status?: number }): string | null {
  if (r.ok) return null;
  const msg = (r.data?.error ?? '').toLowerCase();
  if (msg.includes('abort') || msg.includes('timeout')) return 'timeout';
  if (msg.includes('econnrefused') || msg.includes('connection refused')) return 'connection refused';
  if (msg.includes('enotfound') || msg.includes('getaddrinfo')) return 'unreachable';
  return r.data?.error ?? 'unavailable';
}
// 응답: apiServer: true, marketBot, engineRunning, details: { apiServer: { reason: null }, marketBot: { reason }, engine: { reason } }
```

### 4-2. Discord status embed 반영 (discord-operator)

- **buildStatusEmbedFromApi**: 두 번째 인자 `services`에 `details?: { apiServer?, marketBot?, engine? }` 추가.
- **표시**: `fmt(ok, reason)` → ok이면 `🟢`, 아니면 `🔴` + (reason 있으면 ` (reason)`). 기존 한 줄 문자열 유지.

```ts
const fmt = (ok: boolean, reason: string | null | undefined) =>
  ok ? '🟢' : '🔴' + (reason ? ' (' + reason + ')' : '');
// 서비스 상태 필드: api-server ${a} · discord-op ${d} · market-bot ${m} · engine ${e}
```

### 4-3. memory delta 로그 (api-server)

- **lastMemorySnapshot**: setInterval 바깥에 `null`로 두고, 매 주기마다 현재 `process.memoryUsage()`와 비교.
- **delta 계산**: `lastMemorySnapshot !== null`이면 rss/heapUsed/heapTotal/external 차이를 MB로 계산해 `+NMB` 또는 `-NMB` 문자열로 붙임. 첫 로그는 `(first)`.
- **갱신**: 로그 출력 후 `lastMemorySnapshot`을 현재 값으로 갱신.

### 4-4. port retry count 로그 (api-server)

- **메시지**: `Port ${PORT} already in use. retry #${listenRetryCount} in ${sec}s` (sec = LISTEN_RETRY_MS/1000).
- **초기화**: `onListenSuccess()` 내부에서 이미 `listenRetryCount = 0` 수행 중.

## 5. 왜 이 수정이 운영에 도움이 되는지

| 수정 | 효과 |
|------|------|
| **reason** | Discord/API로 상태 볼 때 “market-bot 🔴”만으로는 원인 모름 → “timeout”/“connection refused”/“stopped” 등으로 즉시 판단 가능. |
| **memory delta** | 절대값만으로는 상승 추이 파악 어려움 → 5분마다 +N MB로 leak 의심 구간을 빠르게 좁힘. |
| **retry #N in Xs** | 포트 충돌 시 몇 번째 재시도인지, 몇 초 뒤 재시도인지 로그만으로 확인. PM2 로그·대시보드와 함께 사용 시 유리. |

## 6. 이번 턴에서는 설계만 남긴 항목

| 항목 | 내용 |
|------|------|
| **Promise.allSettled** | services-status 내부에서 한 쪽만 실패해도 나머지 결과+reason을 그대로 쓰려면 allSettled 전환 검토. 현재는 Promise.all + deriveReason으로 보완. 주석으로만 반영. |
| **TTL 캐시** | 반복 호출 부담 완화를 위해 1~2초 TTL(lastFetchedAt / cachedStatus) 검토 가능. 이번 턴은 미구현, 주석만. |
| **timeout metrics** | timeout/연결 실패 횟수 집계 시 in-memory counter(예: servicesStatusTimeoutCount) 추가 가능. 메트릭 대확장 없이 설계 포인트만 주석. |

---

## 7. 운영상 주의사항

- **details**는 기존 클라이언트가 무시해도 되고, discord-operator는 새 필드만 선택적으로 사용하므로 호환 유지.
- **memory lastMemorySnapshot**은 프로세스당 1개만 유지하며, setInterval 한 번만 등록해 동작 방해 없음.
- **reason**은 사용자 노출용 짧은 문자열이므로, 상세 디버깅은 기존 error 로그/PM2 로그 활용.
